# OCEAN 人格测评系统 · 人格画像和人格分析与改进方案

> **编写日期**：2026-07-15
> **涉及文件**：`backend/app.py`、`agent_prompt.md`、`agent_qa_prompt.md`、`web/index.html`、`web/report.html`、`题库与赋分标准.md`

---

## 一、总体改进说明

测试暴露了五个核心问题，分别涉及：**AI 赋分的语义理解能力不足**、**质量标记与置信度缺乏区分度**、**人格报告未能引用用户的个性化回答**、**答疑 Agent 语气机械且缺乏上下文感知**、**报告页人格分析文案模板化缺乏感染力**。

这五个问题有一个共同的根因：当前系统在设计上**将用户的 E 选项个性化回复只用于"评分"这一环，评完之后信息就丢失了**——报告和答疑 Agent 都看不到这些内容。同时，评分 Prompt 缺乏对网络用语的认知，置信度机制也过于依赖 LLM 自行判断而缺乏规则层的校准。报告文案方面，每个维度每个等级只有一套固定内容，缺乏变化和共情力。

改进方案的核心思路是：

1. **打通数据流**：让 E 选项回复从"一次性的评分输入"变成"可被报告和答疑 Agent 反复引用的用户画像组成部分"
2. **增强 Prompt 的语义理解能力**：在评分六步法中新增"语义复杂性检测与解锁"步骤，让 LLM 自行识别网络用语、反语、夸张、省略等非字面表达并翻译为标准语义，再基于翻译后的语义进行评分——不依赖静态词表，由 LLM 的语义推理能力覆盖所有情况
3. **规则层 + LLM 层双引擎评分**：在 LLM 评分之前用规则预检测，给置信度和质量标记提供更可靠的基线
4. **重写答疑 Agent 的人设与对话策略**：让它像一位真正的"人格分析师"而不是模板填空机
5. **报告文案全面升级**：从单一固定文案升级为"模板池 + 变量插值 + 智能选择"机制，让每个人看到的内容与众不同且更有感染力
3. **规则层 + LLM 层双引擎评分**：在 LLM 评分之前用规则预检测，给置信度和质量标记提供更可靠的基线
4. **重写答疑 Agent 的人设与对话策略**：让它像一位真正的"人格分析师"而不是模板填空机

## 二、具体改进细节

### 问题 1：AI 对网络用语/流行语/反语等复杂语义识别能力不足

#### 1.1 问题表现

测试中用户回答"我是社交恐怖分子来着"（网络用语，意为"我在社交方面极度恐惧/不擅长"），AI 将其标记为 `off_topic`、置信度 0.00、触发二次评分后仍给 1.0 分。这说明 AI 完全不理解"社交恐怖分子"这个网络用语的含义，将其字面理解为"恐怖分子"从而判定为答非所问。

类似问题也出现在"我选择当场去世"（第14题，语气夸张，AI 同样判定为 off_topic）。

#### 1.2 为什么不用静态词表

静态词表有三重缺陷：

1. **永远滞后**：网络用语以周为单位迭代，今天收录"社恐"，明天可能是"社恐10级"，后天可能是某个新梗。词表永远在追。
2. **无法消歧**：同一个词在不同语境下意思不同。比如"摆烂"在"我好想摆烂"中是放弃努力，在"我这学期摆了太多烂了"中则可能是自责（反映尽责性）。词表看不懂语境。
3. **覆盖范围天然有限**：反语、夸张、比喻、省略等非字面表达的方式是无限的，无法穷举。

#### 1.3 改进方案：在评分六步法中嵌入"语义复杂性检测"步骤

**核心思路**：不靠外部字典，而是在评分 Prompt 的核心人格倾向提取（第二步）之前，插入一个**语义预处理步骤**。让 LLM 自己在评分时先做一轮语义解锁——识别是否存在网络用语、反语、夸张、比喻、自创表达等非字面表达，如果有，先翻译为与人格维度相关的标准语义，再进行后续评分。

需要修改的文件：`backend/app.py` 的 `build_grading_prompt()`、`agent_prompt.md`、`题库与赋分标准.md`。

**A. 修改评分 Prompt：在"第一步：情境锚定"和"第二步：核心人格倾向提取"之间插入语义预处理步骤**

将原来的六步法改为**七步法**，新增"第二步：语义复杂性检测与解锁"：

```python
- 检测结果：[有语义复杂性 / 无语义复杂性]
- 检测到的类型：[网络用语 / 反语 / 夸张 / 省略]（可多选）
- 语义翻译：将用户文本中所有非字面表达翻译为与人格维度相关的标准语义描述。保留原文中无需翻译的部分。
- 翻译示例：
  * 原文"我是社交恐怖分子来着" → 翻译"我在社交场合极度恐惧和不自在"
  * 原文"我选择当场去世" → 翻译"我会感到极度尴尬和不适，想要立刻离开那个场合"

- 翻译后的语义才是评分的真正输入。人格倾向提取应基于翻译后的语义，而非原文字面。
```

**C. 将语义解锁结果纳入 confidence 计算**

语义复杂性本身不是坏事——用户用网络用语或夸张表达来回答是正常的。但**语义解锁的确定性**会影响置信度：

```
在第七步（质量标记 + 置信度）中，增加以下置信度调整因子：

与语义解锁相关的调整：
+0.05  语义表达直接清晰，无需解锁（白话文，无网络用语/反语/夸张）
+0.00  检测到网络用语/夸张但语义翻译明确，翻译后含义清晰
-0.05  存在反语或省略表达，语义翻译需要一定推理
-0.10  存在多重语义复杂性（如同时有网络用语+反语），翻译可能不够确定
-0.15  语义解锁困难，多种翻译方向都有可能，无法确定真实含义
       → 此时应同时将 quality_flag 标记为 weak_match 或 boundary
```

### 额外收益：语义解锁对其他问题的连带改善

这个改动不仅解决网络用语识别问题，还会自然改善其他三个问题：

- **对问题 2（置信度单一）**：语义解锁的难度本身就是置信度的合理信号——直白的回答置信度高，含反语的回答置信度自然应该低。LLM 需要显式推理语义复杂度，不再一刀切给 0.80。
- **对问题 3（报告无引用）**：语义解锁后，系统对用户原文的理解更准确，报告中引用时也更容易匹配到正确的解读。
- **对问题 4（答疑机械）**：答疑 Agent 在拿到经过语义解锁的 E 选项数据后，能更准确地理解用户本意，回复自然更贴切。


### 问题 2：质量标记（quality_flag）和置信度（confidence）缺乏区分度

#### 2.1 问题表现

测试的 25 题中，23 题 quality_flag 为 `valid`，仅 2 题在触发二次评分后变为 `off_topic`。23 题 valid 的置信度几乎全是 0.80，偶尔有一题 0.90。这种"一刀切"的输出意味着 LLM 在自由裁量时倾向于给出一个安全的中间值，失去了质量标记和置信度的设计初衷——区分不同质量的回答。

#### 2.2 改进方案

需要修改的文件：`backend/app.py` 的 `score_with_llm()` 和 `build_grading_prompt()`。

**A. 在 LLM 评分前增加规则层预检测（规则层 + LLM 层双引擎）**

不依赖 LLM 自行判断质量，而是先用规则做预检测，将检测结果作为 hard constraint 传给 LLM：

```python
def precheck_open_text(text: str) -> dict:
    """规则层预检测，在 LLM 评分前执行。"""
    import re

    # 去除标点和空白后的纯汉字
    pure_chinese = re.sub(r'[^一-鿿]', '', text)
    char_count = len(pure_chinese)

    # 有效汉字（去除"啊""嗯""哦""哈"等语气词后的字数）
    filler_chars = set('嗯啊哦哈呵嘿嘻呜呀哇啧诶')
    meaningful_chars = [c for c in pure_chinese if c not in filler_chars]
    meaningful_count = len(meaningful_chars)

    # 敷衍检测
   敷衍_patterns = ['随便', '不知道', '无所谓', '都可以', '没什么', '不太清楚', '就这样']
    is_敷衍 = any(p in text for p in 敷衍_patterns) and meaningful_count < 8

    # 关键词密度检测
    all_keywords = (INTENSIFY_STRONG + INTENSIFY_MEDIUM + INTENSIFY_WEAK +
                    WEAKEN_STRONG + WEAKEN_MEDIUM + WEAKEN_WEAK + MIXED_WORDS)
    keyword_chars = sum(len(kw) for kw in all_keywords if kw in text)
    keyword_density = keyword_chars / max(len(text), 1)

    # 内部矛盾检测（简单规则）
    intensify_count = sum(1 for kw in INTENSIFY_STRONG + INTENSIFY_MEDIUM if kw in text)
    weaken_count = sum(1 for kw in WEAKEN_STRONG + WEAKEN_MEDIUM if kw in text)
    has_contradiction = (intensify_count >= 1 and weaken_count >= 1)

    # 基础置信度（由规则层计算）
    base_confidence = 0.70  # 起点

    if meaningful_count >= 30:
        base_confidence += 0.10
    elif meaningful_count >= 15:
        base_confidence += 0.05
    elif meaningful_count < 5:
        base_confidence -= 0.15
        return {
            "force_flag": "off_topic",
            "force_score": 2.5,
            "confidence_cap": 0.40,
            "reason_suffix": " [规则检测: 有效字数不足5个, 判定为敷衍回答]"
        }

    if is_敷衍:
        base_confidence -= 0.15
        return {
            "force_flag": "off_topic",
            "force_score": 2.5,
            "confidence_cap": 0.45,
            "reason_suffix": " [规则检测: 检测到敷衍表达]"
        }

    if keyword_density > 0.40:
        base_confidence -= 0.20
        return {
            "force_flag": "contradictory",
            "confidence_cap": 0.50,
            "reason_suffix": " [规则检测: 关键词密度过高, 可能存在刻意填充]"
        }

    precheck_result = {
        "meaningful_char_count": meaningful_count,
        "keyword_density": round(keyword_density, 3),
        "has_contradiction": has_contradiction,
        "rule_base_confidence": min(1.0, max(0.0, base_confidence)),
    }

    # 轻度标记
    if meaningful_count < 15:
        precheck_result["suggest_weak_match"] = True

    return precheck_result
```

**B. 将规则预检测结果注入评分 Prompt**

在 `build_grading_prompt()` 中的"第六步"之前加入：

```
=== 规则层预检测结果（优先参考） ===
- 有效汉字数: {meaningful_char_count}
- 关键词密度: {keyword_density}
- 初步矛盾检测: {has_contradiction}
- 建议基础置信度起点: {rule_base_confidence}

以上预检测结果仅供参考。如果你认为用户回答确实质量更高或更低，
可以在预检测基础上调整 quality_flag 和 confidence，但调整幅度不应超过 ±0.15，
且需要在 reason 中说明调整理由。
```

**C. 增加 confidence 的精细化调整因子**

在评分 Prompt 中替换原有的简单 confidence 调整规则：

```
置信度精细化调整（在预检测的 rule_base_confidence 基础上应用）：

加分项（每一项最多生效一次）：
+0.08  核心人格倾向非常明确，且有一致的行为描述支撑
+0.05  回答包含具体的生活场景或经历（不是抽象描述）
+0.05  回答长度 ≥ 30 个有效汉字且关键词密度 < 8%
+0.03  回答展现自我反思或自我觉察（如"我发现自己..."）

减分项：
-0.10  锚点匹配在两级间摇摆，难以确定唯一档位
-0.08  回答中有矛盾信号（既想社交又想独处，且未给出情境区分）
-0.05  回答过于简短（< 10 个有效汉字）
-0.12  回答包含多个无关联话题，核心倾向难以聚焦
-0.15  回答中的关键词密度 > 30%

最终 confidence = rule_base_confidence + sum(调整项)
限制在 [0.0, 1.0]，且 quality_flag 为 off_topic/contradictory 时 confidence 上限为 0.50。
```

**D. 后处理：规则层对 LLM 输出做合理性校验**

在 `score_with_llm()` 中，收到 LLM 返回后，检查 LLM 的 confidence 是否合理：

```python
# 后处理校验
if precheck.get("force_flag"):
    # 规则层已判定为低质量，强制覆盖
    quality_flag = precheck["force_flag"]
    final_score = precheck.get("force_score", final_score)
    confidence = min(confidence, precheck["confidence_cap"])
elif abs(confidence - precheck.get("rule_base_confidence", 0.7)) > 0.20:

    confidence = round((confidence + precheck["rule_base_confidence"]) / 2, 2)
```

### 问题 5：报告页人格分析文案过于模板化，缺乏感染力与丰富度

#### 5.1 问题表现

当前报告页（`web/index.html`）中 `dimensionData` 和 `overallPortrait` 对每个人格维度每个等级只有**一套固定文案**。不管你测出什么分数组合，报告显示的内容总是完全一样的——同一个"高开放性"的人，每次看到的 tagline 都是「你对世界有一种还没看完的好奇心」，definition 也完全相同。

具体问题：
1. **维度 tagline（一句话概括）**：15 个 tagline（5 维度 × 3 等级）质量尚可，但每个人的报告里只有一套说辞，缺乏新鲜感
2. **维度 definition**：描述偏理性/定义式，缺少故事感和共情力。比如"你不只是想知道是什么，更想知道如果…会怎样"——这句话不错，但同一维度是否可以有多重视角来看？
3. **校园场景（campus）**：三个场景偏"描述行为"，不够生动
4. **gifts / shadows / tip**：有的写得有温度（如"好奇心是你的燃料"），但有的偏平淡
5. **整体画像（overallPortrait）**：拼接方式固定——"你像是一个XX的人。XX；XX。同时你XX，而XX。" 五个句子顺序永远不变
6. **原型名称（archetypeRules）**：只有 7 种原型 + 1 个兜底"平衡探索者"，远不足以覆盖 3⁵=243 种组合
7. **AI 建议（aiSuggestions）**：只有一套静态内容，不根据实际分数变化

**B. 选择机制：如何从变体池中选取最合适的文案**

```javascript
function selectVariant(dimensionKey, level, results, eResponses) {
    const pool = DIMENSION_VARIANT_POOL[dimensionKey][level];
    if (!pool || pool.length === 0) return pool?.[0];

    // 策略 1：如果用户在该维度有 E 选项回复，优先使用匹配的视角
    const dimEResponses = (eResponses || []).filter(r => r.dimension === dimensionKey);
    if (dimEResponses.length > 0) {
        // 分析 E 选项中的语义特征来匹配变体
        const eText = dimEResponses.map(r => r.userResponse).join(' ');
        for (const variant of pool) {
            // 如果用户表达了犹豫或矛盾，选"温和"视角
            if (variant.flavor.includes('温和') && 
                /犹豫|纠结|有时候|可能|不确定/.test(eText)) {
                return variant;
            }
            // 如果用户表达强烈，选"极致"视角
            if (variant.flavor.includes('探索') && 
                /喜欢|享受|非常|绝对|完全/.test(eText)) {
                return variant;
            }
        }
    }

    // 策略 2：根据分数精细化选择
    const avg = results[dimensionKey].avg;
    if (level === '高' && avg >= 3.7) {
        // 极端高分 → 选"极致"视角
        return pool.find(v => v.flavor.includes('探索者')) || pool[0];
    }
    if (level === '高' && avg < 3.3) {
        // 刚刚达到高分 → 选"温和"视角
        return pool.find(v => v.flavor.includes('温和')) || pool[0];
    }

    // 策略 3：轮选（同一用户每次刷新时给不同变体，但同一次报告中保持一致）
    const seed = Object.values(results).reduce((s, r) => s + Math.floor(r.avg * 10), 0);
    const index = seed % pool.length;
    return pool[index];
}
```

```javascript
function generateOverallPortrait(results, eResponses) {
    // 识别主导维度和次要维度
    const dims = ['O', 'C', 'E', 'A', 'N'];
    const sorted = dims.map(d => ({ key: d, avg: results[d].avg, level: results[d].level }))
        .sort((a, b) => b.avg - a.avg);

    const top = sorted[0];      // 最突出的维度
    const second = sorted[1];   // 次突出
    const bottom = sorted[4];   // 最不突出的维度
    const mid = sorted[2];      // 中等维度

    // 叙事模板池（3-4 套，根据 top 和 bottom 维度选择）
    const narratives = [];

    // 模板 1：主导维度驱动型
    narratives.push({
        condition: () => true,
        generate: () => {
            const topDesc = DIM_HIGHLIGHT_PHRASES[top.key][top.level];
            const bottomDesc = DIM_HIGHLIGHT_PHRASES[bottom.key][bottom.level];
            return `如果只用一个词来形容你的性格轮廓，那大概是「${ARCHETYPE_TRAITS[top.key][top.level]}」。` +
                `${topDesc.opening} ` +
                `${second.avg >= 3.0 ? `而在${DIM_NAMES[second.key]}这件事上，你同样有自己独特的节奏——${secondDesc.bridge}。` : ''}` +
                `不过有意思的是，${bottomDesc.contrast}` +
                `${eResponses && eResponses.length > 0 ? generateEReflection(eResponses) : ''}`;
        }
    });

    // 模板 2：矛盾与张力的叙事
    narratives.push({
        condition: () => Math.abs(sorted[0].avg - sorted[4].avg) >= 1.5,
        generate: () => {
            return `你的人格画像里有一个很有趣的张力：` +
                `一方面，${DIM_HIGHLIGHT_PHRASES[top.key][top.level].contrastIntro}；` +
                `另一方面，${DIM_HIGHLIGHT_PHRASES[bottom.key][bottom.level].contrastIntro}。` +
                `这两个看似矛盾的特质，在你的日常里其实是这样共存的——` +
                `${generateTensionNarrative(top, bottom, eResponses)}`;
        }
    });

    // 根据条件选择模板
    const chosen = narratives.find(n => n.condition()) || narratives[0];
    return chosen.generate();
}
```

#### 5.3 文案升级的迭代方式

考虑到 5 个维度 × 3 个等级 × 3-5 套文案变体 ≈ 60-75 段文案需要撰写，建议分两阶段实施：

**阶段一（本次迭代）**：每个等级先写 2 套变体（30 段），确保有两个不同视角可选。优先覆盖 5 个维度中分数出现频率最高的等级组合。

**阶段二（后续优化）**：根据用户测试反馈，在用户最常看到的等级上扩充到 4-5 套变体，并让文案更口语化、更有故事感。

## 四、验证方式

改进完成后，建议进行以下验证：

1. **语义解锁 - 网络用语测试**：用"社交恐怖分子""社恐10级选手""纯纯社牛""i人实锤"等进行测试，确认语义解锁步骤正确翻译为与人格维度相关的标准语义（如"极度恐惧社交""内向""外向"），并据此给出合理评分而非 off_topic
2. **语义解锁 - 反语/讽刺测试**：用"我社交超厉害的（白眼）""对对对我就爱团队合作（敷衍脸）""你可太了解我了呵呵"等反语表达进行测试，确认解锁步骤能识别反语并翻转语义
3. **语义解锁 - 夸张/自嘲测试**：用"当场去世""社死一万次""我会裂开""天塌了"等进行测试，确认能提取夸张背后的核心情感而非字面理解
4. **语义解锁 - 混合语义测试**：用同时包含网络用语+夸张+反语的复杂回答测试（如"表面笑嘻嘻心里mmp，我社交超牛的，当场社死那种"），确认解锁步骤能逐层拆解且 confidence 会因此下调
5. **置信度区分度测试**：分别用认真长回答（>30字白话文）、简短敷衍回答（<5字）、语义复杂的网络用语回答、矛盾回答进行测试，确认不同情况的 confidence 有明显区分（如直白回答 0.85+、含反语回答 0.60-0.70、敷衍回答 ≤0.40）
